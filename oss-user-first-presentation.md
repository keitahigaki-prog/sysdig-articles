# 1. 問題提起
## 共通の違和感

---

## よくある課題：ベンダー主導の限界

### 営業視点
- 「顧客の要望が開発に届かない」
- 「ロードマップが不透明で説明できない」
- 「ベンダーロックインへの懸念を払拭できない」

### 技術者視点
- 「ブラックボックスで中身が見えない」
- 「カスタマイズやバグ修正を待つしかない」
- 「コミュニティの知見を活用できない」

---

## ユーザーの声が届かない構造

```
顧客要望 → 営業 → プロダクトマネージャー → 開発チーム
         ⬇         ⬇                  ⬇
       変換      優先度判断           実装
       (情報劣化)  (ブラックボックス)   (数ヶ月〜年単位)
```

**結果**: タイムラグ、情報の非対称性、期待値のずれ

---

## 問い：もっと良い方法はないか？

- ユーザーと開発者が直接つながる
- 透明性のある開発プロセス
- コミュニティの知恵を活用できる
- ベンダーに依存しない選択肢

→ **オープンソースソフトウェア（OSS）駆動のアプローチ**

---

# 2. OSSとは何か
## 抽象から具体へ

---

## OSS（オープンソースソフトウェア）の定義

### 4つの自由
1. **使用の自由**: どんな目的でも使える
2. **研究の自由**: ソースコードを読んで学べる
3. **再配布の自由**: コピーを共有できる
4. **改変の自由**: 改良して公開できる

### キーワード
透明性 | コラボレーション | コミュニティ | 持続可能性

---

## OSSのビジネスモデル

### 商用OSSの代表例
- **Red Hat**: Linux（RHEL）→ IBM買収（340億ドル）
- **MongoDB**: データベース → 時価総額200億ドル超
- **HashiCorp**: Terraform, Vault → IBM買収（63億ドル）
- **Elastic**: Elasticsearch → 時価総額60億ドル超
- **Sysdig**: Falco → エンタープライズランタイムセキュリティ

---

## 商用OSSモデルの仕組み

```
        OSS（無償・コミュニティ版）
               ⬇
    ユーザー採用・検証・フィードバック
               ⬇
  商用製品（エンタープライズ機能・サポート）
               ⬇
        収益 → OSS開発へ再投資
               ⬇
          好循環サイクル
```

**Win-Win**: ユーザーは選択肢、ベンダーは持続可能性

# 3. FalcoとRuntime
## クラウドネイティブセキュリティのOSS

---

## Falcoとは

### Cloud Native Computing Foundation (CNCF) の卒業プロジェクト
- **Kubernetes** と同じCNCFファミリー
- **ランタイム脅威検知・レスポンス**のデファクトスタンダード
- Sysdig社が2016年にOSSとして公開

### 主な機能
- システムコール監視によるリアルタイム脅威検知
- カスタマイズ可能なルールエンジン
- Kubernetes環境との深い統合

---

## Falcoの仕組み

```
┌─────────────────────────────────────┐
│   アプリケーション / コンテナ        │
└─────────────────────────────────────┘
              ⬇ システムコール
┌─────────────────────────────────────┐
│         Linuxカーネル                │
│    ┌──────────────────────┐         │
│    │ eBPF / カーネルモジュール│         │
│    └──────────────────────┘         │
└─────────────────────────────────────┘
              ⬇ イベントストリーム
┌─────────────────────────────────────┐
│          Falco Engine               │
│   ルールマッチング・脅威検知         │
└─────────────────────────────────────┘
              ⬇ アラート
      SIEM / 通知 / 自動対応
```

---

## Falcoの検知例

### ユースケース
- **不審なシェル実行**: コンテナ内での`/bin/bash`起動
- **機密ファイルアクセス**: `/etc/shadow`の読み取り
- **ネットワーク異常**: 予期しない外部接続
- **特権操作**: コンテナからのホストマウントアクセス

### 実際のルール例
```yaml
- rule: Terminal shell in container
  desc: A shell was spawned in a container
  condition: spawned_process and container and shell_procs
  output: "Shell in container (user=%user.name container=%container.name)"
  priority: WARNING
```

# 5. ユーザーファーストの構造
## OSSがもたらす好循環

---

## 従来型 vs OSS駆動の比較

### プロプライエタリ（従来型）
```
顧客 → ベンダー → 製品
     （一方通行）  （ブラックボックス）
```

### OSS駆動
```
ユーザー ⇄ コミュニティ ⇄ ベンダー
   ⬇           ⬇          ⬇
  検証      コード貢献    製品化
   ⬇           ⬇          ⬇
フィードバック ← 改善 ← エンタープライズ機能
```

**双方向**: 透明性、スピード、共創

---

## ユーザーファーストを実現する5つの要素

### 1. **透明なロードマップ**
   - GitHubのPublic Issueで機能要望・バグ管理
   - 誰でも閲覧・コメント可能

### 2. **直接的なフィードバックループ**
   - Slack/Discordでリアルタイムコミュニケーション
   - Issue → PR → マージが数日〜数週間

---

## ユーザーファーストを実現する5つの要素（続き）

### 3. **選択の自由**
   - OSSのまま利用継続可能
   - 商用版への移行もスムーズ
   - ベンダーロックインなし

### 4. **コミュニティの知恵**
   - 世界中のエキスパートの知見
   - ベストプラクティスの共有
   - 問題解決の速さ

---

## ユーザーファーストを実現する5つの要素（続き）

### 5. **エコシステム**
   - サードパーティ統合（Prometheus, Splunk等）
   - プラグインアーキテクチャ
   - 標準化されたインターフェース

---

## 実例：機能リクエストから製品化まで

### ケーススタディ
1. **ユーザーがGitHub Issueで機能要望**
   例: "AWS ECS Fargateサポートが欲しい"

2. **コミュニティでの議論・設計**
   → RFC（提案）の作成、技術的実現性の検証

3. **コントリビューションまたはSysdig開発**
   → PRマージ、次のリリースに含まれる

4. **OSS版リリース → 商用版にも反映**
   → すべてのユーザーが恩恵を受ける

**期間**: 数週間〜数ヶ月（従来の1/5〜1/10の速度）

---

## ユーザーファーストのビジネス効果

### 営業視点
- ✅ 顧客の要望を「コミュニティで検討中」と具体的に説明可能
- ✅ ロードマップの透明性で信頼獲得
- ✅ OSSでのPoC → 商用版への自然な移行

### 技術者視点
- ✅ 内部構造を理解した上で導入判断
- ✅ 自社でカスタマイズ・拡張可能
- ✅ コミュニティからの問題解決サポート

# 6. 締め
## 進化し続ける製品

---

## OSSがもたらす継続的進化

### なぜOSS駆動は強いのか

1. **コミュニティの多様性**
   - 様々な業界・ユースケースからのフィードバック
   - ベンダー単独では思いつかないアイデア

2. **品質の担保**
   - 多くの目によるコードレビュー
   - 実際の本番環境での検証

3. **長期的な持続可能性**
   - ベンダーに依存しないライフサイクル
   - CNCFのようなファウンデーションによるガバナンス

---

## Sysdig & Falcoの今後

### ロードマップ（公開情報）
- **AI/ML統合**: 異常検知の高度化
- **マルチクラウド対応**: AWS, Azure, GCP横断の統合管理
- **eBPF活用拡大**: より低オーバーヘッドな監視
- **自動修復（Auto-remediation）**: 検知から対応まで自動化

### 日本市場での展開
- エンタープライズサポートの強化
- 日本語ドキュメント・UIの拡充
- ローカルパートナーエコシステムの構築

---

## まとめ：OSS駆動だからこそのユーザーファースト

### 3つのキーメッセージ

1. **透明性**: ブラックボックスではなく、見える・触れる・貢献できる

2. **スピード**: ユーザーの声が数週間で製品に反映される

3. **選択の自由**: ベンダーロックインのない、持続可能な選択肢
